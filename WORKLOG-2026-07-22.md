# Worklog — Wednesday, 22 July 2026

## Theme
Rolled out an **in-page + API module-gating system** for the CRM so each company variant (V1–V4) only sees the add-on modules it's entitled to. Variant 4 remains untouched (full feature set); V1/V2/V3 see progressively fewer add-ons. Base lead→deal pipeline never breaks.

> Note: No git commits were made today — all work below is in the working tree (uncommitted). Last commit was `359e574` on 20 Jul (Seed DemoAdmin + service analytics dashboard).

---

## What was done (morning → evening)

### 1. Schema + session plumbing
- **`prisma/schema.prisma`** — added `enabledModules String @default("[]")` to `Company` model (stores the module list per company as JSON).
- **`lib/auth.ts`** — added `enabledModules?: string` to `TokenPayload` so the JWT carries the company's modules.
- **`components/AuthProvider.tsx`** — added `enabledModules` to `UserProfile` and the nested `company` object so client components can read it.

### 2. Module registry (single source of truth)
- **`lib/config/moduleVariantMap.ts`** (new) — defines `MODULE_KEYS` (16 add-on modules) and `getModulesForVariant(v)` returning the cumulative module list per variant tier. Mirrors the existing `isVariant2/3/4` sidebar logic so behavior stays identical until cutover.
- **`lib/config/moduleEnforcement.ts`** (new) — `ENFORCED_MODULES` set + `isModuleEnforced()`. Currently set to **all modules enforced** (the single per-module rollout/rollback switch).

### 3. Helpers
- **`lib/modules.ts`** (new) — client-safe `hasModule()` with **fail-open fallback**: if `enabledModules` is missing/malformed, falls back to variant-based logic (byte-identical to today's sidebar). Also `logShadowGuard()` for shadow-mode warnings.
- **`lib/moduleGuard.ts`** (new) — server-only `enforceModuleGuard()` for API routes. Returns a 403 `NextResponse` if the module is enforced and the subject lacks it; otherwise logs a shadow warning and returns null (caller continues).

### 4. Backfill script
- **`scripts/backfill-enabled-modules.ts`** (new) — computes `Company.enabledModules` from the existing `Company.variant`. Dry-run by default; `--apply` to write. Skips companies that already have a non-default value (never overwrites manual customization).

### 5. UI gating components
- **`components/ModuleGate.tsx`** (new) — three exports:
  - `<ModuleGate>` — wraps children; renders a lock overlay + upsell link to `/settings` if the module is missing.
  - `<LockedSection>` — keeps the section header visible (so users know the feature exists) but replaces the body with a lock overlay.
  - `useHasModule()` — hook for client components.

### 6. Sidebar cutover (Phase 7)
- **`app/(dashboard)/layout.tsx`** — wired the sidebar to `hasModule()` via a `moduleSubject` built from the auth user. Module-gated nav items now use `hasMod(MODULE_KEYS.X)`; `isVariantX` is still used for sub-item depth (not module access). (+123 / −75 lines.)

### 7. API route guards (89 routes)
Wired `enforceModuleGuard()` into **89 API route handlers** (+530 insertions) across every add-on module:
- `approvals`, `catalogue` (brochures/categories/datasheets/products/specs/bulk-import-export), `competitors` (incl. lost-analysis, win-loss), `deals`, `documents`, `forecast`, `key-accounts` (relationships/visits/upcoming-reviews), `negotiations` (notes/request-approval/revisions), `purchase-orders` (reconcile/sync-erp), `rfq` (assign-costing/costing-sheet/documents/generate-quotation/line-items/reopen/requirement-report/bulk/customer-submit/export/parse-template/stats), `samples`, `system-config/allow-direct-quotation`, `targets` (achievement), `territories` (accounts/performance), `visits` (attachments/attendees/checkin/checkout/complete/customer-unavailable/reschedule/key-account-compliance).
- Pattern per handler: `const guard = enforceModuleGuard(user, MODULE_KEYS.X, "VERB /api/..."); if (guard) return guard;` placed right after the auth check.

### 8. Plan document
- **`docs/module-gating-plan.md`** (new) — the full phased implementation plan (Phases A–G) with module→variant mapping table, per-page gating spec (opportunity detail, quotation detail, deal detail, customer master), server-side stage-advance fix, implementation order, and a testing checklist.
