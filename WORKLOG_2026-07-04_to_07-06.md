**Key findings (code-confirmed, not guesses)**:

1. **Dual Deal-status vocabularies on one field** â€” `Deal.status` is written by two incompatible
   value sets: Pipeline stages (`Qualified/RequirementGathering/MeetingScheduled/DemoConducted/Rejected/Lost`)
   vs. Deals module values (`Active/OnHold/Won/Lost`). Confirmed via `lib/module-status-config.ts`
   comments plus divergent write sites (`updateDealStatusAction` normalizes only Pipeline values;
   `POST /api/quotations/[id]/create-deal` and `.../accept` write `"Active"`/`"Won"` directly,
   bypassing `transitionDealStatus()`).
2. **ERP sync is entirely manual** â€” no cron job, webhook, or status-change hook triggers
   `POST /api/purchase-orders/[id]/sync-erp`. It requires PO `status === "Approved"` and two env
   vars (`SUKI_ERP_API_URL`, `SUKI_ERP_API_KEY`); on failure, retry is manual (re-call the same endpoint).
3. **Hardcoded API-key fallback** in the inbound lead-ingestion endpoint (`POST /api/leads`):
   falls back to literal `"suki_secret_key_123"` if no `SystemConfig`/env value is set.
4. **No scheduler config in the repo** for any `app/api/cron/*` route (no `vercel.json` cron,
   no GitHub Actions cron workflow) â€” invocation mechanism for these routes is external/unconfirmed.
   A separate standalone `scripts/email-scheduler.ts` (`node-cron`) exists for follow-up email
   summaries and visit auto-checkout, but only runs if started as its own long-lived process.
   *(Note: today's `5f6ac72` commit added `app/api/cron/rfq-autoclose/route.ts` and
   `scripts/lead-sla-scheduler.ts` â€” worth re-checking whether these close this gap.)*
5. **Inbound leads get no auto follow-up task** â€” the auto-follow-up creation block in
   `POST /api/leads` is explicitly commented out (`// DISABLED`), unlike the manual `createLeadAction` path.
6. **Two distinct "copy quotation" endpoints** (`clone` vs `duplicate`) with different code formats
   (`QT-YYYY-NNNNN` vs `QUO-NNNN`) and different revision-tracking behavior.
